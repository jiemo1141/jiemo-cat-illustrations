# 介墨小黑猫怪诞正文配图

为中文文章设计和生成 16:9 横版正文配图。目标不是做商业插画、PPT 信息图或可爱卡通，而是把文章里的关键判断、流程、结构、状态或隐喻，变成一张清爽、怪诞、有创意、可读但不说明书的手绘解释图。

默认视觉 IP 是"小黑猫"：黑色圆润团子身体、黄色大眼睛、粉色小鼻子、有腮红、表情丰富。小黑猫必须参与画面的核心动作，不能只是站在旁边当装饰。

## 先读这些参考

按任务需要读取，不要一次塞满上下文：

- `references/style-dna.md`：风格 DNA、颜色、文字、禁忌（与 Ian 版一致）
- `references/cat-ip.md`：小黑猫 IP 的形象、性格、动作库和禁忌
- `references/composition-patterns.md`：结构类型、原创隐喻方法和反复刻规则（与 Ian 版一致）
- `references/prompt-template.md`：单张生图提示词模板（猫版）
- `references/qa-checklist.md`：生成后检查和迭代规则（猫版）

## 交付模式：用户自选

这个 skill 支持两种交付模式，在 Step 2 出 shot list 时确认用户走哪条：

- **模式 A — 提示词交付（默认）**：我只写提示词，用户自己去其他平台（ChatGPT/DALL·E/Midjourney 等）生成图片。适用于当前模型无图生能力，或用户想自己控制生成质量。
- **模式 B — 自动生成**：我直接用图生工具生成并保存到本地。适用于已配置图生后端的运行环境。

**重要：除非用户明确要求"帮我生成"，否则默认走模式 A。** 模式 A 下跳过图片生成步骤，改为交付提示词。

## 工作流

### 1. 消化正文

先读用户给的正文、链接、Notion 页面、Markdown 文件或截图内容。提炼：

- 核心观点是什么
- 哪些段落承担认知转折
- 哪些内容适合用图解释
- 哪些地方只适合文字，不需要图

不要平均配图。优先选择"认知锚点"，例如：核心判断、两个断点、输入输出闭环、分流、前后对比、一鱼多吃、承接路径、常见坑、角色状态变化。

### 2. 先出配图策略

不管哪种模式都先出 shot list。每张图写清楚：

- 放在哪个段落后
- 图的主题
- 核心意思
- 结构类型
- 小黑猫在图里做什么
- 建议元素
- 建议中文标注词

默认 4-8 张。文章很短时 1-3 张；长文也不要轻易超过 9 张。够用就好。

用户确认 shot list 后，确认走模式 A 还是模式 B。

### 3. 单张生成

#### 模式 B — 自动生成

用图生工具每张单独生成。不要把多张图拼在一张里。

每张图的提示词必须包含 prompt-template.md 里的完整模板内容，替换变量为当前文章的具体内容。

默认 aspect ratio 为 16:9 横版。

不要复刻过往案例。案例只提供风格密度和小黑猫参与方式，不能直接复用已有构图，除非用户明确要求复刻某张图。每次都要从当前文章重新发明一个奇怪但成立的隐喻。

#### 模式 A — 提示词交付（默认）

为每张图生成一个完整的提示词，直接套用 `references/prompt-template.md` 的模板。
提示词是"即用型"——用户拿到就能直接粘贴到 DALL·E / Midjourney / 其他图生工具。
每张图的 prompt 单独列出，不要拼成一张。

### 4. 检查与迭代（模式 B 专用）

只有模式 B（自动生成）需要这一步。生成后用 vision_analyze 查看生成的图片，对照 qa-checklist.md 检查。如果出现以下问题，优先重生成：

- 小黑猫只是装饰
- 画面太满
- 太像流程图/PPT
- 中文太多或错字严重
- 左上角出现"常见坑/流程图/系统架构图"等标题
- 画风太死板
- 背景不是干净白底

### 5. 保存交付

- **模式 A**：在回复中列出每张图的完整提示词（每张一个独立代码块），附上 shot list 说明每张图的位置、主题和核心意思。
- **模式 B**：把最终图片保存到 `assets/<article-slug>-illustrations/` 目录，按顺序命名。

## 输出口径

- **模式 A**：先出 shot list → 用户确认 → 交付完整提示词。shot list 要短而准，提示词要完整可直接复制。
- **模式 B**：先出 shot list → 用户确认 → 生成 → 检查 → 交付图片。交付要包含生了几张、每张用途、保存路径、哪些最稳。

不要长篇解释风格理论；让图自己说话。
